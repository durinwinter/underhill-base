 6                                        INTRODUCTION
 7   This standard specifies general concepts for the automated integration of intelligent equipment,
 8   called Process Equipment Assemblies (PEAs), in higher-level automation systems, called
 9   Process Orchestration Layers (POL) and provides the basic definitions for the specification of
10   interfaces, information models and workflows. The term of “process” in Process Equipment
11   Assembly and Process Orchestration Layer refers to all kind of processes independent of any
12   concrete industry, e.g., process, logistics or manufacturing industry.

13   These concepts can be used for the automation of modular production systems as well as for
14   the integration of automated units into conventional plants (package units). The focus of this
15   specification is on the automation of modular production systems in different industries.
16   Driving forces behind this progress are…
17          … enhancing the efficiency due to reusing and reutilizing not only engineering efforts
18           but as well physical equipment,
19          … increasing flexibility and adaptability        of   production systems   due to the
20           exchangeability of equipment and
21          … reducing time-to-market due to the ability to combine pre-automated, pre-tested and
22           possibly pre-qualified equipment.
23   The engineering workflow of modular production systems considers both the automation
24   technology of the Process Equipment Assemblies, called PEA engineering, and their integration
25   into a Process Orchestration Layer, referred to as POL engineering. This series of standards
26   defines and specifies the Module Type Package (MTP), which semantically describes the
27   interfaces and their functions of the automation technology of PEAs thus enabling the
28   integration of the PEAs into a POL.
29


30   This standard part focuses on the following aspects:

31          basic concept of automation technology for Modular Plants and modular equipment
32          conceptual architecture and structure of the MTP considering various aspects
33          definition and description of the MTP Manifest
34          definition and description of the MTP packaging conventions
35          definition and description of the MTP AttachmentSet
36          definition and description of the MTP TextSet

39                    Part 1: General Concepts, Interfaces and Models
40   1   Scope

41   This document specifies the basic conceptual elements for the Module Type Package concept.

42   In the series of specifications for the Module Type Package concept, interfaces, information
43   models and workflows are specified in order to be able to implement the use cases for the
44   automation of Process Equipment Assemblies and, in particular, production systems in all
45   industries that work with automated systems. It is important to note, that this series of standards
46   does explicitly not specify implementation specific aspects of the PEA and POL software. The
47   focus is on the automation interfaces, their functionality and semantic description.

48   The stakeholders of this series of specifications are Automation Suppliers for PEAs as well as
49   POL solutions, PEA manufacturer, System Integrators and Plant Owners/Operators.

50   For automation suppliers with focus on PEAs these documents provide assistance with regard
51   to the necessary information content and the MTP export interface for their integrated
52   development environments.

53   For automation suppliers with focus on POLs these documents provide semantic and syntactic
54   information to be able to automate the integration of PEAs into their POL products.

55   For PEA manufacturers, these documents provide information on how to improve the integration
56   of their products. They can provide an easy-to-use interface and description that allows their
57   PEAs to be integrated into POLs faster, easier and with fewer errors, giving them an advantage
58   over competitors who do not offer an MTP-compliant product.

59   For system integrators, these documents provide information about the functionality of PEAs,
60   which supports the selection of the right equipment for the required functionality. The functional
61   description within the MTP provides important information, which are used by the system
62   integrator to integrate several PEAs into a running production system.

63   For plant owner/operators, these documents provide information models and meaningful
64   definitions that can be used to formulate functional requirements for a desired PEA and make
65   it possible to compare PEAs already available on the market to determine whether they are
66   suitable for the desired applications.

67   It is possible that several stakeholders are represented by one and the same person or
68   organization - e.g., a plant owner/operator can also be the PEA manufacturer of its own PEAs.

69   Modularization of production systems has gained prominence across industries, spawning
70   various research initiatives and definitions. Process Equipment Assemblies (PEAs) are
71   classified as pre-automated integrable units with dedicated automation systems and integration
72   interfaces. Following a 'graybox' approach, PEAs encapsulate internal monitoring data while
73   maintaining minimal inter-PEA dependencies and self-protection mechanisms. Future
74   developments may enable PEAs to communicate directly, facilitating both orchestrated and
75   choreographed functions while optimizing instrumentation.

76   More details about PEAs, POLs and the general concept of the MTP are provided in Section 7.
77   The contents of other parts of this specification are outlined in Section 8.

214   7     Concept of the Module Type Package

215   7.1    Introduction and Conceptional Overview

216   Modularization represents a significant step towards accelerating the construction and planning
217   of plants. By using industry-specific Process Equipment Assemblies (PEAs) that form ready-
218   made, self-contained functional units consisting of the mechanical construction, automation
219   hardware and software, the effort to plan and build plants can be significantly reduced. The
220   reuse of proven PEAs avoids errors in early phases, which in turn leads to cost and time
221   savings. In addition to the desired scalability through the addition and removal of PEAs
222   (numbering up), a better data basis is also created for cost estimation for the planning and
223   construction of the plants, as usual efforts are merely repeated in a new configuration.

224   This concept of constructing and planning a plant leads to significantly changed phases in the
225   engineering and operation which are visualized in Figure 1. The engineering process is
226   separated into the engineering of single PEAs (PEA engineering) and the assembly of a Modular
227   Plant by integrating PEAs into a POL (POL engineering). At the end of the PEA engineering
228   phase an MTP file is created, which semantically describes the automation interfaces, the
229   provided Services and other information of the PEA. This MTP file enables the automation of
230   the PEA integration during the POL engineering.




231


232            Figure 1 – Engineering phases of Modular Plants (source: NAMUR/ZVEI)

233   The phases of the PEA manufacturer (PEA engineering) and PEA integrator / plant operator
234   (POL engineering/operation) are strictly decoupled from each other. The added value for PEA
235   manufacturer lies in the multiple application of a PEA design; the added value for PEA
236   integrator/plant operator lies in the reduced project time for the entire plant.

237   7.2    Phases for PEA Integrators and Plant Operators

238   If several PEAs are combined and physically connected to form a plant, it becomes necessary
239   to integrate these PEAs into a POL within the framework of POL engineering (Figure 1).

240   During the import of the several PEAs the POL must ensure that all objects of a PEA to be
241   integrated are clearly distinguishable within the context of the POL, for example by assigning a
242   PEA-specific prefix.




      © Copyright PNO, NAMUR, ZVEI 2024 – All Rights Reserved                    Page 18 of 86 Pages
                   Draft for Member Review. Do not Claim Conformance!

243   Basically, the process of Orchestration covers the entire engineering process of composing and
244   integrating PEAs, recipe creation, commissioning and finally operating and monitoring the plant.
245   It is divided into POL engineering and POL operation.

246   POL Engineering

247   POL engineering consists of design, planning, implementation, simulation, and validation of an
248   orchestrated production environment. The concrete steps to be executed during POL
249   Engineering vary in different industries. In the engineering phase the term “Orchestration” is
250   used to describe the linking of PEAs into a meaningful processing facility. This includes for
251   example a topology, workflows and interlocks, the overarching logic, signals, parameters,
252   naming, and versioning.

253   During POL engineering, the PEAs are integrated into the POL (Figure 1, right-hand side).
254   Finally, the PEA description is imported into the POL via the MTP, and the necessary
255   configuration is generated within the functional subsystems of the POL for each required PEA.
256   This includes both the interface to the information model of the PEA and the digital
257   representation of it in the various functional subsystems of the POL. For example, the digital
258   representation of a PEA in the functional subsystem HMI (Human Machine Interface) consists
259   of process displays to be used to monitor and operate the PEA.

260   In addition, the PEA-spanning procedural process is configured in the POL. The procedure
261   control for the timely retrieval and monitoring of the Automation Services of the PEAs is created.
262   The control of the Services of the PEAs is parameterized and, if applicable, interlocking logic
263   across PEAs is generated.

264   POL Operation

265   In the POL operation (Figure 2), the plant is commissioned in its entirety with its PEAs. For this
266   purpose, the physical communication is mapped and parameterized during the provisioning.
267   Previously created recipes and Services can be started. The operating screen aspects of the
268   Modular Plant and the PEAs are displayed with corresponding dynamic values. The plant and
269   thus the running process can be operated and observed within the operation.

274   7.3   Phases for PEA manufacturers

275   PEA Engineering

276   Each PEA is developed once (Figure 1, left-hand side). This includes both the physical design
277   of the PEA and the information technology interface to higher-level systems. In addition, there
278   is the engineering of the control logic and the human-machine interfaces. The engineering
279   corresponds to that of a small plant, with the difference that the PEA should be designed
280   generically in its function for different use cases. The data generated in engineering about the
281   structure, the information interfaces provided by the PEA, the Services, the associated process
282   displays and alarms are stored in the MTP.

283   7.4   Role of the Module Type Package

284   Key of the Module Type Package concept is the package itself which enables the automated
285   integration of the PEA into the POL. As indicated within Figure 1, it contains information out of
286   the PEA engineering, which are mandatory to be available within the POL engineering.

287   In the past, such an information exchange was carried out with proprietary formats like PDF
288   and Excel documents as shown in Figure 3. With the MTP all communication information as
289   well as the information for POL functionality like visualization, reporting, archiving, alarming
290   and other semantical described contents allows a completely automated integration process.
291   This enables an avoidance of transcription errors and errors caused by lack of concentration
292   due to monotonous engineering work.

298   standardized interface elements, communication mappings, semantically enriching information

301   8     List of MTP Specification Parts

302   This section provides a list of all currently available MTP specification parts (except from the
303   one on hand), which are available to implement the integration of a PEA into various functional
304   subsystems of the POL. The concepts, interfaces and model elements for each functional area
305   are specified in one individual part of the MTP specification. In the current version of the
306   specification the following parts are available.

307   8.1     MTP Specification – Part 2: Process Displays

308   Within this part of the MTP specification, the concepts and model elements are specified, which
309   are necessary to monitor and control PEAs from the central HMI system within the POL. The
310   challenge regarding the operability and observability of the process distributed over several
311   PEAs is both the automatic generation of process displays, and the implementation of a uniform
312   “look-and-feel” of a plant HMI.

313   Since the PEA manufacturer is responsible for the planning, construction, and programming of
314   the PEA, he designs the process displays using the objects defined in this and the other parts
315   of this specification series. However, at this point, the manufacturer does not have knowledge
316   of the operating picture library of the POL that is usually used project-specifically in industrial
317   plants. The generation of the PEA-specific process displays in the POL of the entire plant can
318   therefore only take place after integration of the PEA in the POL engineering. During the import
319   into the POL, the generic process displays objects are replaced by concrete, target system-
320   specific symbols.

321   For the implementation of the PEA-specific process displays in project-specific standardized
322   process display objects, the process displays must be available in a representation-independent
323   description form. This description contains information about the type of process equipment to
324   be displayed as well as its position and size. This information can be evaluated by an algorithm
325   which places the project-dependent control screen elements in the desired representation and
326   position on the HMI in the POL and links them to the corresponding variables for communication
327   with the PEA control system. The use of a library concept requires that the library is available
328   both in the engineering tool of the PEA manufacturer and in the engineering tool for the POL.

329   All information developed in PEA engineering and required during POL engineering is stored in
330   the MTP. The corresponding information models are defined in the MTP Specification – Part 2.

331   8.2     MTP Specification – Part 3: Base Interfaces

332   This part of the MTP specification specifies the concepts, base interfaces, and the model
333   elements, which are necessary to access dynamic data like status information and process
334   values of a PEA from the POL.

335   Usually, process displays of PEAs or the entire plant use symbols and faceplates for sensor
336   and actuator objects. To be able to transport state and status information or to control them
337   manually, interfaces are specified for those objects.

338   First base interfaces for control modules like valves, drives and controls, as well as indicators
339   or manual operations are specified. Furthermore, some blueprints for complexity reduction are
340   described for the interfaces of MTP Specification Part 3.

341   The control module DataAssembly definitions of MTP Specification Part 3 are used in MTP
342   Specification Part 4 as well to be referenced as required equipment.

343   8.3   MTP Specification – Part 4: Automation Services and Process Values

344   With part 4 of the MTP specification, three new building blocks of the MTP concept are
345   introduced.

346   Automation Services, which act as procedural function abstractions, are used in sequential
347   execution control engines within the POL and represent one of the most important conceptual
348   parts from the MTP concept. The state-based behavior behind the procedural abstraction with
349   the possibility to define multiple procedures as different variants of process functions,
350   parameterization and reporting capabilities completes the Service concept.

351   To implement cross-PEA control loops or interlocking logic in the POL, the concept, interfaces,
352   and model elements for process value interconnections are specified within this part, too. This
353   release of the MTP specification foresees the interconnection of process values from different
354   PEAs within the POL.

355   With the introduction of Automation Services, the use of Service interaction mechanisms and
356   status information of Services requires human readable multi-language-based texts which shall
357   be shown within the POL visualization. Specification part 4 introduces the models to be used
358   for those Service-specific text definitions based on the general text definitions introduced in the
359   specification part on hand.

360   8.4   MTP Specification – Part 5: Runtime and Communication

361   Over the last years a lot of different communication technologies came up and provide benefits
362   in different scenarios and situations. To be future-proof the MTP concept in general foresees a
363   technology-agnostic description format for the communication aspect.

364   To be usable within the MTP concept, each communication technology needs a technology-
365   specific specification to define which capabilities provided in the MTP concept in general are
366   mappable to the specific technology. Otherwise, workarounds or unusable features are a
367   consequence.

368   Part 5 provides some fundamental specification parts as well as a list of topics which have to
369   be specified dependent on each used technology in their technology-specific specification.

370   8.5   MTP Specification – Part 5.1: Runtime and Communication – Open Platform
371         Communications – Unified Architecture

372   The standard Open Platform Communications – Unified Architecture (OPC UA) provides wide
373   communication capabilities based on an object-oriented address model accessible in a client-
374   server architecture pattern.

375   This part of the specification defines the minimum conformance units of the OPC UA
376   specification which shall be available to be used within the MTP concept. The different
377   communication features foreseen in the MTP concept are mapped to OPC UA mechanisms and
378   some standardized address model constructs are introduced.

379   Additional to the basic communication mappings, an improved provisioning workflow next to the
380   standard workflow is described following the possibilities of OPC UA.

381   Which parts of the OPC UA specification are used within the MTP concept and how they are
382   used, is described in MTP Specification Part 5.1.

393   9     Principles and Conventions for the Module Type Package Modelling

394   9.1     Principles and Conventions for AutomationML Definitions

395   9.1.1     Using of Attributes fields Value and DefaultValue

396   If an attribute has a value, the field value shall be used by default. If there are any exceptions
397   from this general statement and the general AML definition, it is mentioned explicitly within the
398   specification.

399   In general, the field DefaultValue may be used. Mandatory use of the field DefaultValue is
400   mentioned explicitly as well.

401   9.1.2     Mandatory use of Object ID of all AutomationML Objects

402   The AutomationML Property ID of all InstanceHierarchies, InternalElements, and
403   ExternalInterfaces shall be set to a unique GUID-formatted value (RFC 4122). Other
404   AutomationML objects may set the ID Property. If other Aspects require an ID reference to
405   AutomationML objects which do not specify a mandatory ID, this will be stated in the according
406   document.

407   9.2     Principle Model Architecture and Package Format

408   The MTP uses an open architecture using AutomationML 2.10 (Automation Markup Language,
409   see IEC 62714) with the underlying file format CAEX 3.0 (Computer Aided Engineering
410   Exchange).

411   Each MTP is packed following the Open Packaging Conventions as described in Section 14.

412   The manifest of an MTP is an AutomationML file containing at least three InstanceHierarchy
413   (IH), namely ModuleTypePackage, DataAssemblies, and ServerAssemblies. Further parts of
414   the specifications may define further aspects that are referenced from the MTP manifest.
415   Thereby, each aspect shall be represented by a separate IH in the manifest. Existing aspects
416   are not affected by this, so that aspects can be gradually included in the MTP in the context of
417   further parts of the MTP Specification.

418   9.3     Aspect-oriented Model Extensions and their Model Definitions

419   The corresponding versions of the individual aspects defined in the MTP Specifications,
420   according to which the contents of the MTP are structured, shall be specified in the MTP file.
421   The versioning shall be done according to the definitions in Section 10.

422   For each aspect in the MTP, a corresponding SystemUnitClassLibrary (SUCL) is created, which
423   contains the classes necessary for modelling the aspect.

424   If ExternalInterfaces (EI) are required, then an additional InterfaceClassLibrary (ICL) will be
425   defined for the respective aspect.

426   If optional description components are required, these will be modelled in the form of
427   RoleClasses (RCs), thus a RoleClassLibrary (RCL) is created, if required. It must thereby be
428   explicitly defined whether the RCs are to be defined as SupportedRoleClasses (SRC) or
429   RoleRequirements (RR).

430   If AttributeTypes (AT) are required, an AttributeTypeLibrary (ATL) will additionally be developed
431   for the respective aspect.

521   10 MTP Versioning

522   This section specifies the versioning of and within the MTP concept.
523   The concept foresees…
524           … a versioning of the specification documents,
525           … a versioning of the definitions within the specification and
526           … a versioning of and within the package and the corresponding PEA system.
527   Within Section 10.1 the versioning of the specification documents is defined. Within
528   Section 10.2 a basic versioning scheme is described, which shall be applied for all content
529   related versioning information. Section 10.3 introduces the concept of aspects and profiles for
530   versioning the specified content of the MTP concept. In Section 10.4 the representation of the
531   versioning is specified. Further information regarding versioning is given within the aspect-
532   specific sections.
533   10.1    Versioning of the Specification Documents

534   This standard consists of several parts, which focus on different topics necessary for the
535   integration of PEA into the POL. To maintain the consistency within the overall standard, the
536   publication of the documents follows a release cycle concept.

537   For internal references, the individual documents shall be identified by the title and the year
538   (YYYY) of publication (e.g., "ModuleTypePackage Part <Number>: <Title>, <Year>” or “MTP
539   Part <Number>:<Year>” for short). If it is the same released document date, the year suffix may
540   be removed – “MTP Part <Number>”.

541   For external references, the individual documents shall be identified by title, the year of
542   publication and the document identifier from PI (e.g. “MTP Specification Part <number>, <Title>;
543   <Year> PI Specification no. 30.<document identifier>”). If the entire specification shall be
544   referenced, the title, the year of publication and the generic document identifier is used (e.g.
545   “MTP Specification; <Year>; PI Specification no. 30.X”).

546   10.2    Versioning Scheme

547   For the content-related version information, a common scheme is used. The versioning shall
548   be notated in the Major-Minor-Patch Notation, specified in [1], which is formatted as follows:

549   Major-Minor-Patch Notation
550   Notation scheme: {Major}.{Minor}.{Patch}

551   The major version is identified by the main version number. A new major version indicates
552   significant and breaking changes to the aspect under consideration. Backwards compatibility to
553   older major versions is rarely given.

554   The minor version is identified by the minor version number. A new minor version indicates an
555   extension of the aspect under consideration. Backwards compatibility to older minor versions
556   is given under certain conditions.

557   The patch version is identified by the patch version number. A new patch version indicates
558   minor bug fixes of the aspect under consideration. Backwards compatibility to older patch
559   versions is given to unlimited extent.

560   To model this version information, the abstract AT SemanticVersionAttributeType is introduced
561   in Section 11.7.15.

562   10.3     Concept of Aspects and Profiles

563   The concept of aspects and profiles introduces the versioning mechanisms of the specification
564   content. In general, an aspect focuses on a defined functional part or subsystem of the POL.
565   Within such an aspect, related definitions for DataAssemblies, models, workflows, and
566   constraints are grouped to so-called profiles. Each aspect shall have a base profile to ensure
567   the interoperability on a minimum level of standard functionality. Further profiles may be used
568   to extend different aspects use case- or industry-specifically and to provide an easy way to
569   introduce new conceptual topics. Extension profiles are complementary and shall not replace,
570   change or invalidate the base profile. The relations between the package, aspects, profiles, and
571   definitions are shown in Figure 4.




572


573       Figure 4 – Conceptional relations between the Package, its Aspects, profiles and
574                       individual definitions in the MTP Specification

575   10.3.1     Aspect Models

576   Different aspect models are specified throughout the different parts of this standard. An aspect
577   model groups all DataAssembly definitions, model elements, and constraints to describe all
578   relevant content to integrate the PEA into one functional subsystem of the POL.

579   10.3.2     Profiles

580   A profile is a collection of related definitions for DataAssemblies, model elements and
581   constraints of one aspect. The scope of a profile shall be limited to one aspect. Profiles shall
582   not overlap, i.e., each definition for DataAssemblies, model elements, or constraints specified
583   in this standard belongs to exactly one single profile.

584   To be compliant to a profile, all relevant DataAssemblies, model elements and constraints shall
585   be supported. Every aspect shall feature at least one profile, which is the “Base”-profile. An
586   aspect may include multiple extension profiles. To ensure interoperability, profiles shall be
587   versioned as described in Section 10.2. The unique identification of a profile consists of the
588   aspect title, profile title and the version. The profile designation shall follow the shown scheme:

589   ModuleTypePackage:{AspectTitle}.{ProfileTitle}:{Version}

608   11.2   Concept of the Manifest and its Table of Content

609   To have a central entry point into the context of the standardized information models of the
610   MTP concept the manifest is introduced. The manifest shall be an IH with the name
611   ModuleTypePackage. Within the manifest, exactly one InternalElement (IE) of the SUC
612   ModuleTypePackage (Table 20) shall be instantiated, with the name of the PEA type. In
613   accordance with IEC 62714-1 this name shall not be empty.

614   A PEA shall have a set of standardized information which, in addition to type and instance-
615   specific information, also enables MTP-PEA verification by means of a type, instance and
616   version comparison. The workflows are described in Section 11.8. The information available
617   offline shall be contained as attributes at the IE of SUC ModuleTypePackage (Table 20). The
618   information available at runtime shall be available in the DataAssembly of SUC
619   PeaInformationLabel (MTP Part 3).
620   Furthermore, the possibility of a web visualization is provided. For this purpose, an interface
621   may be defined using the SUC WebServerUrlInfo (MTP Part 3), from which the web server
622   address can be retrieved at runtime.

623   A main component of the Manifest is the table of contents for the aspects within the MTP. It
624   refers to the different aspects so that easy navigation through the MTP is possible. For this
625   purpose, a SUC MTPSet (Table 21) shall be available, which serves as a parent class, from
626   which classes can be derived that shall be defined in the respective SUCL of further aspects
627   of this standard. These derived SUCs in turn shall be instantiated hierarchically under the IE
628   of SUC ModuleTypePackage by IEs. Each IE whose type is derived from the SUC MTPSet
629   represents an entry in the manifest's table of contents. This is used to refer to the various
630   aspects of the MTP. Within the directory entries there shall always be an instance of an IC
631   AspectSetReference (Table 24) from the ICL MTPICLib, whose attribute AspectRef is to be set
632   to the value (Value) of the ObjectID of the associated IH within the file of the Manifest. The
633   attribute AspectRef of the AspectSetReference inherits the meaning of the AT
634   AspectIDAttributeType from the ATL MTPATLib. The AT AspectIDAttributeType (Table 27)
635   defines the attribute AspectRef of the data type xs:string, which by definition contains the
636   ObjectID of an aspect IH. This AT is derived from the generic AT IDReferenceType (Table 26).
637


638   11.3   Concept of ID Links

639   ID links describe a referencing mechanism that references any AutomationML entity via its
640   ObjectID by means of an attribute value. The value of the attribute contains the ObjectID of the
641   referenced entity. The ObjectID of an AutomationML entity is of the data type xs:string and is
642   to be executed as GUID (Globally Unique Identifier). Thus, the referencing attribute of the
643   AttributeDataType is also to be executed as xs:string. To identify an ID link, the AT shall be set
644   to IDLinkAttributeType (Table 28) of the ATL MTPATLib (Table 25). This AT is derived from the
645   generic AT IDReferenceType (Table 26). ID links are used for references with 1:1 or n:1
      © Copyright PNO, NAMUR, ZVEI 2024 – All Rights Reserved                      Page 38 of 86 Pages
                   Draft for Member Review. Do not Claim Conformance!
      Module Type Package – Part 1: General Concept, Interfaces and Models                Version d0.9

646   relations.
647   The use of the ID link concept is referenced in appropriate sections throughout the remainder
648   of this and other parts of the standards.

649   11.4   Concept of LinkedObjects

650   The LinkedObject concept shall be used to link related partial descriptions from different aspect
651   models of the MTP. The core of the concept is the abstract definition of the SUC LinkedObject
652   (Table 22), as well as the AT RefIDAttributeType (Table 29). This defines how the variable
653   RefID is to be interpreted in the IE of the SUC LinkedObject.
654   The SUC LinkedObject (Table 22) is located in the SUCL MTPSUCLib (Table 19). The SUC has
655   an attribute RefID of the AT RefIDAttributeType defined in the ATL MTPATLib (Table 25). This
656   AT is derived from the generic AT IDReferenceType (Table 26) and defines that the attribute
657   contains a GUID which is used to describe the affiliation of an object to an object of another
658   aspect. For this purpose, the RefIDs of different SUCs inherited from the SUC LinkedObject
659   shall be set to the identical GUID.
660   The class for describing the data objects – SUC DataAssembly – is derived from the SUC
661   LinkedObject, which means that each data object already has the AutomationML attribute RefID
662   and can be referenced to it. If the data objects are referenced from any other MTP aspect, the
663   referencing class shall therefore also be derived from the SUC LinkedObject. This also gives
664   this SUC a RefID and the ability to refer to another LinkedObject by a 1:n relationship.
665   RefIDs of DataAssemblies shall be unique within the MTP, meaning two DataAssemblies shall
666   not share the same value for their RefIDs. In general, a RefID shall not be empty, unless stated
667   otherwise. If RefID is left empty, this indicates that the object is assigned directly to the PEA.

668   11.5   Multilingual Texts

669   Multilingual modelling follows the Best Practice Recommendation [3] for modelling multiple
670   languages within an AutomationML file. The AT MultiLanguageTextAttribute (Table 31) defines
671   the entry point of a multilingual modelling of a text element of the SUC Text. All string variables
672   may be implemented in the form of multi-language modelling if static modelling is used.

673   For each language-specific text, a child attribute shall be added with the corresponding
674   language code according to RFC5646 (e.g. „aml-lang=de-DE“) as name and the language-
675   specific text as Value [3].

787   11.8     Aspect-related Workflow Definitions

788   11.8.1     Overview of the Workflows for Type and Version Verification

789   The workflow specified in this section shall be used to check the affiliation and compatibility of
790   an MTP to a connected, physical PEA. For this purpose, a type and version verification shall
791   be carried out. If both verifications are successfully executed, a clear affiliation and compatibility
792   of an MTP to a PEA can be confirmed. During commissioning of a Modular Plant, the verification
793   described shall be carried out accordingly.

794   The two verification workflows and the necessary DataAssemblies are specified in detail below.

795   11.8.2     Type Verification Workflow

796   The workflow specified in this section shall be used to check the affiliation between a physical
797   PEA and the associated MTP on type level. For this check, the two attributes ManufacturerUri
798   and ProductCode shall be used, which together allow an identification of the PEA type. These
799   two attributes shall be available both in the runtime interface of a PEA and in the MTP. The
800   type identification workflow requires that in the MTP the manufacturer and type information are
801   entered in the attributes ManufacturerUri and ProductCode of the SUC ModuleTypePackage
802   (Table 20).
803
804   The runtime information shall be provided in the attributes ManufacturerUri and ProductCode
805   of the SUC PeaInformationLabel (MTP Part 3). For verification, the two attributes
806   ManufacturerUri (runtime and MTP) shall be checked for equality. The same can be done for
807   the attributes ProductCode. If both comparisons are successful, the MTP describes exactly the
808   type the PEA is of.

809   However, the version may differ, which will be discussed in the following subsection.

810   11.8.3     Version Verification Workflow

811   To check the compatibility of the version of the MTP against the version of the PEA, the
812   following four revisions shall be used:

813   ModuleTypePackageRevision
814   The version information of the entire MTP is part of the SUC ModuleTypePackage, in which the
815   version information shall be entered in a major-minor-patch notation in the Version attribute.
816   For this attribute, the AT ModuleTypePackageRevisionType defined in Table 33 shall be used.

817   DeviceRevision
818   The version of the information technology interface of a PEA is defined as
819   DeviceRevision and shall be implemented in the major-minor patch notation. The information
820   technology interface contains the whole set of information that is modelled in the
821   ServerAssemblySet (MTP Part 5) of the associated MTP. Thus, the DeviceRevision shall cover
822   the sum of all supported protocols and protocol endpoints described in the MTP.

823   The AT DeviceRevisionType defined in Table 34 shall be used for the attributes containing the
824   DeviceRevision.

825   SoftwareRevision
826   The version information of the control software of a PEA is defined as a SoftwareRevision and
827   shall be implemented in the major-minor-patch Notation.

828   HardwareRevision
829   The version information on the constructive structure of a PEA is defined as HardwareRevision
830   and shall be implemented as major-minor-patch notation.

835   The compatibility between a version of the MTP and a PEA shall be checked by means of the

843      Adding an element in the information technology interface shall at least lead to a minor increment
844       of the DeviceRevision.
845      Changing or removing an element in the information technology interface shall lead to a major
846       increment of the DeviceRevision.

850   The following rules for verifying version compatibility shall be applied.

851   Major level

852   A different version at major version level between PEA and MTP indicates an incompatibility of
853   the information technology interface. PEA and MTP are not compatible in this case. If both
854   version specifications are identical, a minor-level check shall be executed to verify full
855   compatibility.

856   Minor level

857   In the following, the prefix PEA is used for the version read from the physical PEA. The prefix
858   MTP stands for the information read from the MTP. At minor level, there are three possible
859   combinations according to the incrementing rules for the DeviceRevision:

860          Case 1 – PEA.DeviceRevision.Minor is identical to MTP.DeviceRevision.Minor
861           If both minor versions are identical, PEA and MTP are compatible.
862

      Module Type Package – Part 1: General Concept, Interfaces and Models                    Version d0.9

863           Case 2 – PEA.DeviceRevision.Minor is greater than MTP.DeviceRevision.Minor
864            If the minor version of the PEA is larger, new content was added to the information technology
865            interface after the MTP was created. No existing content was changed or removed. Thus, the
866            interface of the PEA contains all the interface elements described in the MTP. The new
867            information added to the PEA is just not described in the MTP. The use of the MTP for this PEA
868            implies that new information or functions cannot be used from the POL.
869
870           Case 3 – PEA.DeviceRevision.Minor is smaller than MTP.DeviceRevision.Minor
871            If the minor version on the MTP side is larger, the MTP has new interface information that was
872            not yet loaded on the PEA at the time the version verification was carried out. Thus, the
873            compatibility between PEA and MTP is not given.
874


875   Patch level

876   A change at patch level shall only be used for textual corrections without concept or
877   specification changes. Thus, a change at patch level has no effect on the compatibility between
878   PEA and MTP.

879   11.8.4    Instance Verification Workflow

880   The instance verification workflow is used to determine during commissioning whether the
881   correct PEA instances are used at a specific location within the Modular Plant. During
882   engineering, the PEA instances to be used within the plant are selected from a range of PEA
883   instances. After its activation, the POL shall compare the correctness of the planned setup by
884   comparing the ProductInstanceUri in the interface of the PEA and the offline planning. If it is
885   determined that the online and the offline information are not the same, the operator shall be
886   informed, and appropriate measures shall be offered to react to the discrepancy.

887   11.8.5    Internal Configuration Changes in the PEA

888   The software of a PEA may have several configuration or setting options that may not be
889   accessible via the information technology interface. Reasons for this may be that there are no
890   standardized interfaces for this information, or changeability via the MTP interface is not desired
891   by the PEA manufacturer. In order to make these changes visible via the standardized
892   information interface, a so-called RevisionCounter is introduced, which should be increased in
893   the event of any configuration change within the PEA. By tracking this revision information, the
894   POL can be made aware of a configuration change within the PEA. The RevisionCounter shall
895   be reset to 0 by loading new software, recognizable by a new software revision. The
896   RevisionCounter can only be accessed at runtime. The RevisionCounter attribute defined in the
897   SUC PeaInformationLabel (MTP Part 3) is used for this purpose. A change of the internal
898   RevisionCounter has no effect on the compatibility between PEA and MTP.

983    13.2   Conceptual Aspect Details

984    In general, many concepts within the automation domain are using lists of numbers to express
985    different things. Those lists of numbers are simple for computers but hard for humans to
986    interpret. Goal of the MTP Aspect TextSet is to provide a simple expressive mechanism to add
987    human understandable texts to computer interpretable numbers.

988    In the different parts of the MTP specification a lot of numbers are used to express several
989    information which must be displayed in a clear and sometimes language-dependent human
990    readable look and feel. For exactly those use cases this aspect set provides a base model of
991    TextDefinition and Texts. Texts are used to encapsulate a string which can be provided single
992    or multi-language based and is defined in relation to a concrete value.

993    13.3   Overview of Aspect Definitions

995    standard. With the enumeration definitions one applications of the text model is introduced in

1092   14 MTP Packaging Format

1093   The basis for the packaging format of the MTP are the Open Packaging Conventions (OPC)
1094   specified in ISO/IEC 29500-2, which describe an abstract package model based on packages,
1095   parts, and relationships.

1096   A package describes a container that contains a series of parts (files or resources). Each file
1097   has a unique, URI-compliant name within the package. In addition, the type of each file is
1098   indicated by the MIME type. Relationships can be used to define relationships between the
1099   package itself and individual parts or between parts themselves, the meaning of which is
1100   defined in more detail via a RelationshipType. This enables the definition of a logical model of
1101   the structure or content of packages.
1102
1103   This abstract model of OPC for MTP is defined in the following sub sections.

1104   14.1   Physical Package of the Module Type Package

1105   The mapping to the ZIP format according to ISO/IEC 29500-2 shall be used as the physical
1106   package, so that an MTP file is always transported as a ZIP-compressed file. *.mtp shall be
1107   specified as the file extension for MTP packages. The MIME type of an MTP package shall be
1108   “application/mtp”.

1109   Every MTP package shall contain the AutomationML-based Manifest and its referenced CAEX
1110   XML Schema (XSD) in the root directory of the package. Attachments according to Section 12
1111   shall be stored in a folder with the name “Attachments”. Structuring of attachments into any
1112   substructures within the attachments folder is permitted.

1113   14.2   Logical Model of the Module Type Package

1114   Following the Best Practice Recommendation of the AutomationML Container [4], the following
1115   relationship types and MIME types shall be used:

1116   For the AutomationML-based manifest file within the MTP package:
1117    Relationship type: "http://schemas.automationml.org/container/relationship/RootDocument"
1118    MIME type: "model/vnd.automationml+xml"
1119    File name shall be manifest.aml
1120
1121   For the CAEX Schema of the AutomationML-based MTP file:
1122    Relationship type: "http://schemas.automationml.org/container/relationship/CAEXSchema"
1123    MIME type: "text/xml"
1124    File Name shall be CAEX_ClassModel_V3.0.xsd
1125
1126   For attachments in the “Attachments” folder:
1127    Relationship type: http://schemas.automationml.org/container/relationship/AnyContent
1128    MIME type: is to be set user-defined according to the referenced attachment
1129    File Names shall be defined by the MTP creator
1130
